# Паола App — персональный ассистент (рутина + аналитика)

Новое веб-приложение, закрывающее оба слоя ТЗ:

- **Рутина**: сводка по календарю и задачам, привычки со стриками, напоминания
  (пуш в Telegram), фокус дня.
- **Аналитика**: отдельные чаты по функциям — 🤖 Ассистент, 🇮🇹 Переводчик,
  🔎 Ресерч (веб-поиск с источниками), 🏛 Совет директоров (CFO + CMO + инвестор +
  скептичный клиент-PMI), 🧩 Бизнес-модель. Чаты группируются в **проекты**
  (папки по клиентам): контекст проекта передаётся агенту в каждом чате.

Память — те же базы Notion, что у Second Brain: приложение и бот Паола видят
одни и те же задачи. Код бота (`../bot.py`) не затронут.

## Архитектура

```
static/ (веб-интерфейс)  ⇄  FastAPI (main.py)
                              ├─ agent.py — агентный цикл Claude, 5 режимов чатов
                              ├─ storage.py — SQLite: проекты, чаты, история сообщений
                              ├─ tools/  — задачи · календарь · привычки · напоминания
                              │            переводы · совет директоров · материалы (Notion)
                              ├─ services/ — Notion (async) · Google Calendar · Telegram
                              ├─ digests.py — утренняя/вечерняя сводки
                              └─ scheduler.py — напоминания (каждую минуту),
                                                дайджесты 08:00/21:00 (опционально)
```

**Истории чатов** хранятся в SQLite. Чтобы они переживали редеплой, на Railway
подключи Volume (Service → Settings → Volumes → Mount Path `/data`) и добавь
переменную `DATA_DIR=/data`. Без volume чаты живут до следующего деплоя.

Модели: `claude-sonnet-5` (агент, аналитика), `claude-haiku-4-5` (переводы).

## Подготовка Notion (один раз)

Создать три базы и расшарить их интеграции (той же, что у бота):

**Привычки** (`NOTION_HABITS_DB_ID`)
| Свойство | Тип |
|---|---|
| Привычка | Title |
| Активна | Checkbox |
| Цель | Text (необязательно) |

**Журнал привычек** (`NOTION_HABIT_LOG_DB_ID`)
| Свойство | Тип |
|---|---|
| Запись | Title |
| Привычка | Text |
| Дата | Date |
| Выполнено | Checkbox |

**Напоминания** (`NOTION_REMINDERS_DB_ID`)
| Свойство | Тип |
|---|---|
| Напоминание | Title |
| Когда | Date (с временем) |
| Отправлено | Checkbox |

База «Задачи» — существующая, ID тот же, что у бота.

**Материалы** (`NOTION_MATERIALS_PAGE_ID`) — обычная страница (не база),
например «Материалы Паола App». Сюда Паола сохраняет объёмные результаты
research и разборов из чатов **вне проектов** отдельными подстраницами
(markdown-таблицы становятся настоящими таблицами Notion) и даёт ссылку в чат.
Страницу расшарить той же интеграции (⋯ → Connections), её ID — из URL
страницы (32 hex-символа).

**Клиенты** (`NOTION_CLIENTS_PAGE_ID`) — обычная страница, например «Клиенты
Паола App»; расшарить интеграции так же. При создании проекта внутри неё
появляется **досье клиента** (карточка, цели, факты и решения). Досье — память
проекта: подгружается Паоле в каждый чат проекта, новые устойчивые факты она
дописывает туда сама (`update_client_memory`), материалы проекта сохраняются
подстраницами досье. Чаты внутри проекта работают единым агентом со всеми
инструментами — режим выбирать не нужно.

**Двусторонняя связь проектов ↔ Notion.** Направление работает в обе стороны:
- приложение → Notion: создал проект в приложении → под «Клиенты» появилось досье;
- Notion → приложение: завёл папку под «Клиенты» вручную → она подхватывается
  как проект (и переименования тоже). Синхронизация запускается при открытии
  приложения, при заходе на вкладку «Проекты» и по кнопке ↻ там же. Свои и
  чужие страницы приложение различает по сохранённому `notion_page_id`, так что
  дублей досье не создаёт. Удаление пока не синхронизируется: убранную в Notion
  папку приложение не трогает (проект и его чаты остаются).

## Консунтиво Generali (учёт часов и биллинг)

Учёт часов по клиенту Generali для фактуры. Три базы Notion под страницей клиента
«Generali» (внутри «Клиенты Паола App», доступ интеграции наследуется):
**Consuntivo Generali** (строки-активности A–N с историей с сентября 2025),
**Tariffe Generali** (тип → ставка €/час), **Reparti Generali** (справочник
департаментов: волна, ответственный, фаза). ID заданы дефолтами в `config.py`.

Поток с подтверждением (инструменты в `tools/consuntivo.py`):
1. «Собери консунтиво за июль» → `list_generali_events` тянет встречи за период
   (в т.ч. прошедшие) → агент классифицирует (коды `W#-F#`→Ondata/Fase,
   `CI→CoInd`, `SAL→Track`, `BBR→Progett`, нет кода→Progett) → `add_consuntivo_rows`
   пишет строки **черновиком** (`Stato = TO CHECK`, НЕ в фактуру) и собирает
   отчёт-страницу + пуш в Telegram.
2. Юлия проверяет/правит строки в базе (вид «Da confermare»).
3. «Подтверждаю июль» → `confirm_consuntivo` переводит `TO CHECK → Confermato` —
   только подтверждённое идёт в фактуру.
4. «Сколько часов за июль» → `query_consuntivo` (часы/суммы по репартам и типам).

Правила фильтрации календаря: `Yulia a Monza` (тех. присутствие) скрыто в
`gcal.GCAL_SKIP_TITLES`; личные события без участников `@agmonza.it` не биллятся.

## Локальный запуск

```bash
cd paola-app
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # заполнить значения
uvicorn main:app --reload
# открыть http://127.0.0.1:8000
```

## Деплой на Railway (отдельный сервис, Паолу не трогает)

1. Railway → New Service → тот же GitHub-репозиторий.
2. Settings → **Root Directory: `paola-app`** — сервис видит только эту папку.
3. Variables → скопировать по списку из `.env.example`.
4. Start command подтянется из `paola-app/Procfile`.
5. Networking → Generate Domain — это адрес приложения.

Бот Паола продолжает работать как отдельный сервис из корня репо — они не
мешают друг другу. Когда приложение возьмёт на себя утро/вечер, включить
`DIGESTS_TO_TELEGRAM=1` и отключить соответствующие job'ы у бота.
